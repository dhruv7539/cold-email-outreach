# Cold Email Proof Bank

Use this with [master_data.md](master_data.md) when drafting outreach.

Goal: every email should include one concrete, role-matched proof line instead of a generic stack list.

## Rules

- use `1` proof line per email
- prefer a `metric`
- if a metric is not the best fit, use a `verifiable artifact`
- do not mix three accomplishments into one sentence
- the proof line should answer: `why am I already adjacent to this problem?`

## Contact-Type Guidance

### Recruiter

- use the clearest metric with the least jargon

### Hiring Manager / Engineering Manager

- use the most role-matched systems proof

### Engineer

- use a peer-respect proof: OSS, distributed systems, performance, production debugging

### Executive / Director

- keep it concrete but legible; emphasize impact, not tool names

## Fallback Rule

If there is no strong metric for the role, prefer your strongest artifact or most memorable proof.
